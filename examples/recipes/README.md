# Recipe kitchen ‚Äî a chained, policy-driven workflow with PROV provenance

The example that shows a **multi-step workflow where both Dizzy loops do real work
together**: a chain of three recipes, each producing an entity the next one needs, all
advanced by a single **policy** reacting to **W3C PROV-style events**.

```
active_starter ‚îÄ‚ñ∂ sourdough_loaf ‚îÄ‚ñ∂ garlic_croutons
```

A **recipe** is a template: an ordered list of typed steps, each with its inputs, a
tool, and an output type ‚Äî structured data, never free text. A **batch** is a durable
running instance of a recipe. A batch whose required input entity does not exist yet
opens **blocked**; producing that entity unblocks it.

## The cascade (why this example exists)

Open all three batches up front. The loaf and croutons batches open *blocked* ‚Äî the
entities they require (`active_starter`, `sourdough_loaf`) do not exist yet. Then
advance **only** the starter:

```
advance_batch(starter-1)
  run_batch  ‚îÄ‚ñ∂ step_performed √ó n, entity_produced(active_starter)         (reactivity)
  entity_produced ‚îÄ‚ñ∂ inventory_store + generation_graph                      (data)
  entity_produced ‚îÄ‚ñ∂ advance_ready_batches ‚îÄ‚îÄ(query find_blocked_batches)‚îÄ‚îÄ‚ñ∂ advance_batch(loaf-1)
      run_batch ‚îÄ‚ñ∂ entity_consumed(active_starter), entity_produced(sourdough_loaf), entity_derived
      entity_produced ‚îÄ‚ñ∂ advance_ready_batches ‚îÄ‚îÄ‚ñ∂ advance_batch(croutons-1)
          run_batch ‚îÄ‚ñ∂ entity_produced(garlic_croutons)
          advance_ready_batches ‚îÄ‚ñ∂ nobody waiting; cascade ends
```

Each batch ends by emitting `entity_produced` (prov:wasGeneratedBy). The host folds
that fact into the read models (the **data loop**), then hands it to the
`advance_ready_batches` **policy** (the **reactivity loop**). The policy asks
`find_blocked_batches` who was waiting for that entity type and dispatches
`advance_batch` for each. A policy emits commands only, so the host routes
`advance_batch` back into `run_batch`, whose next `entity_produced` re-triggers the
policy ‚Äî until nobody is waiting. The whole chain runs from one trigger.

## Events that communicate relationships (W3C PROV-O)

The events are not prose; they record provenance relationships. Each maps to a PROV-O
term:

| Event | PROV-O term | Records |
|-------|-------------|---------|
| `step_performed` | `prov:Activity` / `prov:wasAssociatedWith` | a step ran, on a batch, with a tool |
| `entity_consumed` | `prov:used` | an activity used an entity (ingredient or upstream lot) |
| `entity_produced` | `prov:wasGeneratedBy` | an entity was generated by an activity/batch |
| `entity_derived` | `prov:wasDerivedFrom` | an output entity was derived from a source entity |

PROV **Entities** are ingredient lots and intermediate/finished products; **Activities**
are step executions; **Agents** are tools. Following Dizzy's convention, these events
carry explicit reference fields (`entity_id`, `activity_id`, `source_entity_id`, ‚Ä¶)
rather than implicit links, and each carries a timestamp ‚Äî so the event stream *is* the
provenance graph. `trace_provenance` walks the `entity_derived` edges back to the root:

```
garlic_croutons wasDerivedFrom sourdough_loaf
sourdough_loaf wasDerivedFrom active_starter
```

## Steps are data, not text

Recipes, their steps, and each step's inputs are typed rows in the `catalog` model,
built by ordinary commands (`define_recipe`, `add_recipe_step`, `add_step_input`). The
`run_batch` procedure reconstructs a batch's work by **querying** that structured data
(`get_recipe`, `get_recipe_steps`, `get_step_inputs`) and emitting one PROV event per
step and per input ‚Äî there is no free-form recipe text anywhere.

## The components at a glance

- **4 models** ‚Äî `catalog` (ingredients, tools, recipes, steps, inputs), `batches`,
  `inventory` (produced lots + availability), `provenance` (the queryable PROV graph).
- **7 procedures** ‚Äî five catalog recorders, `open_batch` (resolves ready/blocked via a
  query), and `run_batch` (emits the PROV facts).
- **1 policy** ‚Äî `advance_ready_batches`, the cascade engine.
- **11 projections, 7 queries** ‚Äî fold facts into the models and read them back.

> `step_performed` and `entity_consumed` are emitted as immutable PROV facts but only
> the parts `trace_provenance` needs (`generation_graph`, `derivation_graph`) are
> projected ‚Äî a deliberate event-sourcing choice: events are the source of truth, models
> are rebuildable views of them.

## Run it

The generated `lib/python-uv/` is a uv workspace. Sync it once, then run `demo.py`
inside that environment ‚Äî from the repository root:

```bash
uv sync --project examples/recipes/lib/python-uv
uv run --project examples/recipes/lib/python-uv python examples/recipes/demo.py
```

Expected output (abridged):

```
Opening batches:
  opened batch croutons-1 [blocked] (blocked on sourdough_loaf)
  opened batch loaf-1 [blocked] (blocked on active_starter)
  opened batch starter-1 [ready]

Advancing starter-1 (watch the cascade):
  ‚úì starter-1 produced active_starter (starter-1:active_starter)
  ‚úì loaf-1 produced sourdough_loaf (loaf-1:sourdough_loaf)
  ‚úì croutons-1 produced garlic_croutons (croutons-1:garlic_croutons)

Final batch states:
  starter-1: completed
  loaf-1: completed
  croutons-1: completed

Provenance of the croutons (trace_provenance):
  garlic_croutons wasDerivedFrom sourdough_loaf
  sourdough_loaf wasDerivedFrom active_starter
```

## Rebuild it from scratch

`def/` stubs and the `src/` implementations inside `lib/` are **never overwritten**, so
to regenerate them too, remove them.

```bash
# from the repo root
rm -rf examples/recipes/lib

# 1. scaffold def/ LinkML schemas + libconfig.yaml (idempotent; won't clobber edits)
uv run dizzy generate definitions examples/recipes/recipes.feat.yaml examples/recipes

# 2. (human step) fill in attributes in def/*.yaml ‚Äî see the committed files here

# 3. compile LinkML into the gen_def/gen_int type packages under lib/python-uv/
uv run dizzy generate static examples/recipes/recipes.feat.yaml examples/recipes

# 4. split each element into a redistributable runtime package
uv run dizzy generate libraries examples/recipes/recipes.feat.yaml examples/recipes

# 5. (human step) implement the src/ stubs in lib/python-uv/<kind>/<name>/src/
#    ‚Äî see the committed files here, especially procedure/run_batch/ and
#      policy/advance_ready_batches/
```

For the minimal feature and the full three-stage workflow, start with
[`guestbook/`](../guestbook/); for a single policy that queries, see
[`library/`](../library/). Then `dizzy docs authoring`.
